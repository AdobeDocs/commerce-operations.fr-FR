#### Améliorations supplémentaires de la sécurité

Les améliorations de sécurité de cette version améliorent la conformité aux dernières bonnes pratiques en matière de sécurité.

**Nouvelle amélioration de la sécurité pour la version 2.4.7-bêta2**

Cette version introduit un nouveau paramètre de configuration du cache de la page entière qui permet d’atténuer les risques associés à la variable `{BASE-URL}/page_cache/block/esi HTTP` point de terminaison . Ce point de terminaison prend en charge les fragments de contenu sans restriction et chargés dynamiquement à partir des poignées de mise en page Commerce et des structures de bloc. La nouvelle **[!UICONTROL Handles Param]** paramètre de configuration définit la valeur de ce point de terminaison. `handles` qui détermine le nombre maximal autorisé de gestionnaires par API. La valeur par défaut de cette propriété est 100. Les vendeurs peuvent modifier cette valeur à partir de l’option Admin ( **[!UICONTROL Stores]** > **[!UICONTROL Settings:Configuration]** > **[!UICONTROL System]** > **[!UICONTROL Full Page Cache]** > **[!UICONTROL Handles Param]** ). Voir [Configuration de l’application Commerce pour l’utilisation du vernis](https://experienceleague.adobe.com/docs/commerce-operations/configuration-guide/cache/configure-varnish-commerce.html). <!-- AC-9113 -->

**Faits saillants de la sécurité introduits dans la version 2.4.7-beta1**

* **Limitation du taux natif pour les informations de paiement transmises par les API REST et GraphQL**. Les commerçants peuvent maintenant configurer la limitation de débit pour les informations de paiement transmises à l&#39;aide de REST et de GraphQL. Cette couche de protection supplémentaire prend en charge la prévention des attaques par carte et peut éventuellement réduire le volume des attaques par carte qui testent de nombreux numéros de carte de crédit en même temps. Il s’agit d’une modification du comportement par défaut d’un point de terminaison REST existant. Voir [Limite de débit](https://developer.adobe.com/commerce/webapi/get-started/rate-limiting/).

* Le comportement par défaut de la variable [isEmailAvailable](https://developer.adobe.com/commerce/webapi/graphql/schema/customer/queries/is-email-available/) Requête GraphQL et ([V1/customers/isEmailAvailable](https://adobe-commerce.redoc.ly/2.4.6-admin/tag/customersisEmailAvailable/#operation/PostV1CustomersIsEmailAvailable)) Le point de terminaison REST a changé. Par défaut, l’API renvoie désormais toujours la valeur true. Les vendeurs peuvent activer le comportement d’origine, qui consiste à renvoyer la variable `true` si l’email n’existe pas dans la base de données et `false` s’il existe.

Les précédentes versions d’Adobe Commerce 2.4.7 ont été introduites :

* **Opérations d’enregistrement plus rapides des configurations au niveau du magasin pour les déploiements avec de nombreux magasins**. L’enregistrement des paramètres de configuration dans les déploiements comportant plus de 500 boutiques peut prendre du temps. La nouvelle [Configuration asynchrone](https://experienceleague.adobe.com/docs/commerce-operations/performance-best-practices/configuration.html?lang=en#asynchronous-configuration-save-2.4.7-beta1) module active des opérations d’enregistrement de configuration asynchrones en exécutant une tâche cron qui utilise un consommateur pour traiter l’opération d’enregistrement dans une file d’attente de messages. Cela réduit le temps d’arrêt requis pour générer le cache de configuration pour les configurations volumineuses. Les opérations d’enregistrement de configuration ne nettoient plus la variable `config_scopes` cache avant d’écrire des données dans le cache, ce qui réduit également le temps pendant lequel d’autres requêtes sont verrouillées pendant l’écriture des données de configuration.

### Serveur d’applications pour les API GraphQL

Commerce 2.4.7-beta2 introduit un nouveau serveur d’applications PHP pour les API GraphQL qui est implémenté sur une structure PHP Openswoole. Ce serveur d’applications permet à Adobe Commerce de conserver un état parmi les demandes d’API GraphQL de commerce et élimine la nécessité d’amorçage, ce qui se traduit par un débit supérieur, une latence inférieure et une utilisation efficace des ressources pour toutes les API GraphQL. En partageant l’état de l’application entre les processus, les demandes de l’API GraphQL deviennent beaucoup plus efficaces et les réponses de l’API GraphQL sont en moyenne 30 % plus rapides.

La variable `ApplicationServer` met en oeuvre le serveur d’applications PHP. Les développeurs qui souhaitent mettre en oeuvre ce module dans des déploiements sur site doivent également installer et configurer Swoole et Nginx. Les développeurs travaillant avec Adobe Commerce sur des projets d’infrastructure cloud n’ont pas besoin de télécharger ces composants, car l’infrastructure inclut déjà Nginx et OpenSwoole. Les développeurs cloud doivent activer manuellement le serveur d’applications et ne peuvent ni démarrer ni arrêter le module comme les autres services. Adobe Commerce doit être redéployé pour démarrer ou arrêter le module, ce qui nécessite un temps d’arrêt du site.

Voir [Serveur d’applications pour les API GraphQL](https://experienceleague.adobe.com/docs/commerce-operations/performance-best-practices/performance-best-practices/application-server.html) pour les déploiements sur site et dans le cloud.

Il s’agit d’une version bêta du module PHP Application Server pour les API GraphQL. Les déploiements qui incluent des personnalisations et des extensions peuvent ne pas être compatibles avec le module. Les commerçants qui activent le module doivent être prêts à résoudre les incompatibilités potentielles avec les personnalisations et extensions déployées. <!-- ACPT-27 -->

**Améliorations B2B introduites dans la version 2.4.7-beta1**

Adobe Commerce 2.4.7-beta1 inclut de nouvelles fonctionnalités et améliorations pour les guillemets négociables B2B, les mises à jour de l’API REST, ainsi que plusieurs correctifs.

   * La couverture API web améliorée des fonctionnalités de storefront pour le catalogue partagé inclut de nouveaux points de terminaison pour affecter et réinitialiser des actions de prix personnalisées : `POST /V1/sharedCatalog/:sharedCatalogId/assignTierPrices` et `POST /V1/sharedCatalog/:sharedCatalogId/resetTierPrices`.

Pour plus d’informations sur la couverture de l’API REST et GraphQL pour Commerce B2B, voir [API Web Commerce](https://developer.adobe.com/commerce/webapi/) documentation destinée aux développeurs.
